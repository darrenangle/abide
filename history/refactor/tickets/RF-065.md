# RF-065: Standardize Word-Count Measurement On Tokenizer-Based Word Boundaries

- Priority: `P1`
- Status: `done`
- Area: `measurement consistency`
- Origin: `DEGENERATE-INPUT-RESCAN-2026-04-01`

## Problem

The repo already has a shared tokenizer in `preprocessing.py`, but several
word-count verifiers still use raw whitespace splitting. That lets punctuation
tokens count as words, which distorts reward calculations. Reproduction:
`WordCinquain` currently gives a five-line punctuation-only poem a failed but
still high score of `0.742` because lines like `"!!!"` are treated as one word.

## Primary Files

- `src/abide/constraints/lexical.py`
- `src/abide/constraints/shape.py`
- `src/abide/forms/free_verse.py`
- `src/abide/forms/mathematical.py`
- `src/abide/forms/novel.py`
- `tests/constraints/test_lexical.py`
- `tests/forms/test_fibonacci_and_free_forms.py`
- `tests/forms/test_exported_form_surface.py`

## Acceptance Criteria

- Word-count semantics should use tokenizer-based word boundaries instead of raw
  whitespace splitting in the active verifier paths.
- Punctuation-only input must no longer count as satisfying one-word lines in
  forms like `WordCinquain`.
- Existing valid word-count fixtures should keep passing.

## Completed

- Switched the active word-count verifier paths from raw whitespace splitting to
  the shared tokenizer, including lexical word-count constraints, shape-based
  word measurement, `FreeVerse`, `OddEvenDance`, and the mathematical forms
  that measure words per line or stanza.
- Strengthened `FreeVerse` word-bound scoring so punctuation-only lines no
  longer remain the highest failed-score outlier once they are correctly
  counted as zero-word lines.
- Added targeted regressions for punctuation-as-word handling plus a surface
  degenerate-input harness that caps failed exported-form scores on the current
  audit corpus.

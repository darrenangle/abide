# RF-075: Standardize The Repo Onto `uv` And Refresh The Lockfile To The Latest Resolvable Dependencies

- Priority: `P2`
- Status: `done`
- Area: `tooling`
- Origin: `UV-REFRESH-2026-04-02`

## Problem

The repo is only partially standardized on `uv`. The package config and CI use
`uv`, but multiple run scripts still hardcode `/home/darren/miniconda3/...`
paths, several script docstrings still instruct bare `python`, and the README
still advertises `pip install abide`. The lockfile also has not been refreshed
after the recent training/runtime work.

## Primary Files

- `pyproject.toml`
- `uv.lock`
- `README.md`
- `scripts/`
- `tests/integration/`

## Acceptance Criteria

- All repo-owned run paths and examples should use `uv`-managed commands rather
  than hardcoded external Python environments.
- The README should present `uv` as the canonical workflow for this repo.
- `uv.lock` should be refreshed to the latest resolvable dependency set.
- The upgraded environment should pass the repo verification suite or any
  targeted runtime checks required by changed training dependencies.

## Resolution

- Replaced the remaining hardcoded `/home/darren/miniconda3/...` launch paths
  in repo-owned run scripts with `uv run`, and updated script docstrings plus
  the README so the repo now presents `uv` as the canonical workflow.
- Added a regression test covering repo-owned docs/scripts so stale miniconda
  paths and `pip install abide` do not reappear silently.
- Refreshed `uv.lock` against current upstream releases and made the training
  stack explicit in `pyproject.toml` by adding direct `accelerate` and `vllm`
  dependencies instead of relying on the removed `verifiers[rl]` extra.
- Verified the upgraded environment with `uv sync --all-extras`, `uv run ruff
  check src/abide tests scripts`, `uv run mypy src/abide`, and
  `uv run pytest -q -W error` -> `893 passed, 1 skipped`.
- Confirmed the refreshed stack now resolves to `vllm==0.18.1`,
  `trl==1.0.0`, `transformers==4.57.6`, and `verifiers==0.1.10`; stable
  Transformers still does not recognize `gemma4`, so the existing Gemma 4
  preflight/source-install fallback remains intentionally in place.

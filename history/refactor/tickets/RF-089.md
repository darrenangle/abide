# RF-089: Generate and Validate a Codex Spark Full-Surface Corpus Covering Every Exported Abide Form

- Priority: `P2`
- Status: `done`
- Area: `training data`, `form surface`
- Origin: `user request`
- Depends On: `RF-087`, `RF-088`

## Problem

The repo now has a verified Codex Spark writing loop for the hard-form subset,
but the user wants coverage across the entire exported Abide form surface. The
current tooling is still oriented around a small named subset. To make this
useful for future SFT and evaluation work, we need a generalized task builder
and a first validated corpus that contains at least one accepted Codex
Spark-authored example for every exported form that instantiates through the
catalog.

## Primary Files

- `src/abide/training/`
- `scripts/`
- `data/sft/`
- `history/refactor/`

## Acceptance Criteria

- The Codex Spark task builder can target the full exported form surface without
  requiring manual form lists.
- A task sheet is generated for every exported form that instantiates through
  `load_form_instances()`.
- A validated artifact is produced with at least one accepted example for every
  exported form in that surface.
- The ticket records the final exported-form count and the validated count.

## Outcome

- Generalized the Codex Spark task builder so it can target either the hard
  subset or the full exported form surface, with a direct `brief_only` prompt
  mode for mechanically constrained forms.
- Generated the full-surface task sheet at
  `data/sft/codex_spark_full_surface_tasks.jsonl` for every form returned by
  `load_form_instances()`.
- Produced a first validated all-forms artifact at
  `data/sft/codex_spark_full_surface_validated.jsonl`.
- Final full-surface counts:
  - exported forms instantiated through the catalog: `141`
  - tasks generated: `141`
  - validated rows: `141`
  - failures: `0`

## Notes

- The original `gpt-5.3-codex-spark` writing pass hit a provider usage limit
  mid-batch, so the full-surface generation finished on fallback subagents.
- The sweep surfaced two genuine code-side blockers that were fixed as part of
  the work:
  - `CombinedChallenge` catalog defaults were impossible at `num_words=4` with
    the inherited `20`-vowel target; the catalog now uses a satisfiable
    `vowel_count=10`.
  - `Terzanelle` had a 19-line outer contract but a 17-line rhyme-scheme child;
    the rhyme pattern now matches the 19-line approximation the verifier claims
    to check.

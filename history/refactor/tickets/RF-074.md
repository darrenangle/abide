# RF-074: Add No-VLLM TRL Fallback For Gemma 4 Canaries While VLLM Lacks Architecture Support

- Priority: `P2`
- Status: `done`
- Area: `training runtime`
- Origin: `GEMMA4-LIVE-BOOT-2026-04-02`

## Problem

The Gemma 4 E4B model now loads through current Transformers, but the current
`vllm==0.12.0` stack still rejects `Gemma4ForConditionalGeneration`. That
blocks the usual two-GPU server path before training can start. We need a
fallback path that can run a tiny TRL canary without vLLM.

## Primary Files

- `scripts/train_grpo_trl.py`
- `scripts/run_grpo_gemma4_e4b.sh`
- `tests/integration/`

## Acceptance Criteria

- `train_grpo_trl.py` must support a non-vLLM generation path.
- The Gemma 4 E4B runner must default to that fallback until vLLM support is
  available.
- A local Gemma 4 E4B canary must start past the previous vLLM architecture
  rejection.

## Resolution

- Added a non-vLLM generation path to `train_grpo_trl.py`, including CLI flags
  for `--use-vllm` and `--no-vllm`, a shared `GRPOConfig` builder, and smaller
  local-generation canary controls.
- Updated the Gemma 4 model profile and runner defaults so Gemma 4 canaries use
  local generation, `sdpa` attention, and language-model-only LoRA targeting.
- Added a Transformers-support preflight to the Gemma 4 runner and verified a
  real one-step Gemma 4 E4B canary on GPU 0:
  `uv run python scripts/train_grpo_trl.py --model google/gemma-4-E4B-it --prompts 1 --batch-size 1 --num-generations 2 --max-completion-length 256 --no-vllm --output models/grpo_trl_gemma4_e4b_smoke --save-steps 10 --telemetry-every 1 --no-wandb`
  which completed with reward telemetry and saved `models/grpo_trl_gemma4_e4b_smoke/final`.

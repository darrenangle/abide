# RF-076: Restore Gemma 4 vLLM Serving By Reconciling Native Uv Runtime With The Upstream Gemma 4 Image Path

- Priority: `P2`
- Status: `done`
- Area: `training runtime`
- Origin: `GEMMA4-VLLM-UPSTREAM-2026-04-02`

## Problem

The refreshed native `uv` stack now carries `vllm==0.18.1`, but Gemma 4 E4B
still does not serve through the local TRL/vLLM path because the stable
`transformers==4.57.6` build does not recognize `gemma4`. Meanwhile the vLLM
team is publicly showing a Gemma 4-capable Docker image path, so we need to
close the gap and capture a real working path in the repo.

## Primary Files

- `pyproject.toml`
- `uv.lock`
- `scripts/run_grpo_gemma4_e4b.sh`
- `tests/integration/`

## Acceptance Criteria

- A Gemma 4-capable serving path must work again, either natively in the repo
  `uv` environment or through a documented container workflow.
- The repo should encode that path clearly enough that later runs do not depend
  on manual archaeology.
- The result should be verified directly against `google/gemma-4-E4B-it`.

## Resolution

- Replaced the drifting `latest nightly` prep logic with a pinned Gemma 4-capable
  native overlay: `vllm==0.18.2rc1.dev73+g66e86f1db` from
  `https://wheels.vllm.ai/66e86f1dbd565292a253e7d2d6851f65dc4f14ba/...` plus
  the matching `transformers` source ref
  `edaac7db98e34208209fd67d8c66781b8c2e4a53`.
- Restored the Gemma 4 E4B training profile to prefer `vLLM` again and added a
  regression test that the prep script stays pinned to the verified overlay
  instead of silently drifting back to `uv pip install vllm --pre`.
- Verified native Gemma 4 `trl vllm-serve` startup directly against
  `google/gemma-4-E4B-it`, then verified a one-step GRPO smoke run against that
  live server which completed successfully and saved
  `models/grpo_trl_gemma4_e4b_vllm_smoke/final`.

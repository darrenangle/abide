# RF-029: Fix Malformed `PositionalPoem` Catalog Defaults And Fail Early On Invalid Positions

- Priority: `P1`
- Status: `done`
- Area: `form catalog`, `hard forms`
- Origin: `SWEEP-2026-04-01`

## Problem

The catalog currently provides malformed default kwargs for `PositionalPoem`: it passes `[1, 2, 3]` instead of a list of `(position, character)` tuples. That means bulk-loaded forms can instantiate successfully and then raise `TypeError` from `PositionalCharacter.verify()` during normal verification.

## Primary Files

- `src/abide/forms/catalog.py`
- `src/abide/forms/hard.py`
- `tests/forms/`

## Acceptance Criteria

- Instantiating `PositionalPoem` through the catalog must no longer inject malformed defaults.
- `PositionalPoem` must validate `positions` in its constructor and raise a clear exception for invalid input.
- Ordinary verification with default settings must return a `VerificationResult` instead of crashing.
- Add regressions for the catalog path and the early validation path.

## Completed

- Removed the malformed catalog override so `instantiate_form("PositionalPoem")` uses the form's valid built-in defaults.
- Added constructor validation in `PositionalPoem` so malformed positions raise `ValueError` immediately.
- Added regressions for the catalog path and the constructor-validation path.
- Verified with `uv run ruff check src/abide tests scripts`, `uv run mypy src/abide`, and `uv run pytest` -> `607 passed, 1 skipped`.

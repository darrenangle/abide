# RF-081: Replace The Legacy Gemma Trainer With A Prime-RL Gemma 4 E2B Path

- Priority: `P1`
- Status: `done`
- Area: `training reliability`
- Origin: `PRIME-RL-GEMMA4-E2B-2026-04-03`

## Problem

The user requirement is to use the current verifiers-supported RL path rather
than the legacy `vf.RLTrainer` stack. Upstream docs now position `prime-rl` as
the primary RL entrypoint, with `verifiers` environments served to the
orchestrator via `vf.EnvServer`. The current repo still centers the Gemma
verifiers workflow on the legacy trainer and its custom live-weight-sync path,
which is especially brittle when combined with 4-bit weights.

## Primary Files

- `pyproject.toml`
- `scripts/`
- `src/abide/`
- `tests/integration/`

## Acceptance Criteria

- The repo must expose a first-class Gemma 4 E2B training path using the modern
  `prime-rl` architecture instead of the legacy `vf.RLTrainer` path.
- Abide must provide the required `verifiers` environment server interface for
  the new path.
- The new Gemma 4 E2B path must be covered by focused tests and direct local
  verification.

## Progress

- Added `scripts/train_prime_rl.py` as the modern Abide `prime-rl` entrypoint,
  with local-snapshot resolution, generated `rl.toml`, runtime-venv support,
  and an installable Abide env package instead of the older legacy trainer.
- Added `scripts/prepare_prime_rl_runtime.sh` and
  `scripts/run_prime_rl_gemma4_e2b.sh`, plus the `abide-poetry-forms`
  environment package under `src/abide/training/` and
  `src/abide_poetry_forms/`.
- Moved the Gemma 4 prime-rl runtime onto a verified stack in
  `.venv-prime-rl`: `vllm 0.18.2rc1.dev73+g66e86f1db`,
  `transformers 5.5.0.dev0`, and `mistral-common 1.11.0`.
- Added `src/sitecustomize.py` as a subprocess-only compatibility layer for
  current `prime-rl` / `vllm` drift, including Gemma 4 VLM registration, the
  missing `run_api_server_worker_proc` symbol, the newer `build_app(...)`
  signature, and stale serving kwargs that current `vllm` no longer accepts.
- Added focused coverage in `tests/integration/test_abide_poetry_forms.py` and
  `tests/integration/test_train_prime_rl.py` for the installable env package,
  local model resolution, and the Gemma 4 prime-rl config generation.
- Switched the Gemma 4 prime-rl path away from LoRA for now. Current `vllm`
  rejects LoRA on `Gemma4ForConditionalGeneration`, so Gemma 4 E2B now uses
  full-weight filesystem broadcast on the modern path while the upstream LoRA
  support gap remains.
- Fixed the remaining `prime-rl` / `vllm` serving drift in `src/sitecustomize.py`:
  current `OpenAIServingChat` kwargs are now resolved from the live signature,
  `init_app_state(...)` compatibility no longer recurses into prime-rl's stale
  monkeypatch, and the orchestrator client base URL is generated against the
  selected inference port instead of the hardcoded legacy `8000`.
- Hardened the Gemma 4 E2B runner/runtime path for 24 GB cards:
  `scripts/run_prime_rl_gemma4_e2b.sh` now uses a truly conservative smoke
  profile, honors `ABIDE_*` override env vars for prompt/batch/token/seq
  sizing, and `scripts/train_prime_rl.py` now defaults the subprocess runtime
  to `PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True`.
- Switched Gemma 4 prime-rl training from optimizer-only offload to full FSDP
  CPU offload plus activation offloading, which removed the backward-pass OOM on
  the one-step Gemma 4 E2B smoke profile.
- Added monitored process-group cleanup in `scripts/train_prime_rl.py` so the
  runner detects a written `final_summary.json`, gives the underlying `rl`
  process a grace window to exit, and then reaps lingering `prime-rl` / `vllm`
  children without turning a successful smoke into a false failure.
- Direct local smoke `live24` completed end-to-end: the Abide env server became
  healthy, `vllm` loaded `Gemma4ForConditionalGeneration`, the trainer executed
  step `0` under `prime-rl`, `final_summary.json` was written, both GPUs were
  released, and the shell runner exited with code `0`.

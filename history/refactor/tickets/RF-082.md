# RF-082: Replace Placeholder Prime-RL Presets With Realistic Short/Long Canaries

- Priority: `P1`
- Status: `done`
- Area: `training reliability`
- Origin: `PRIME-RL-CANARY-PROFILES-2026-04-07`

## Problem

The modern Gemma 4 E2B `prime-rl` path now works end to end, but its built-in
runner presets still reflect plumbing smoke defaults rather than real poem
lengths. In particular, the default smoke path uses `max_tokens=64`, which is
only enough to prove process startup and not enough for the current
`well_known` mix that includes sonnets, villanelles, ghazals, and sestinas. The
current `canary` and `soak` presets are also placeholder values rather than
validated budgets tied to an explicit form subset.

## Primary Files

- `scripts/run_prime_rl_gemma4_e2b.sh`
- `scripts/train_prime_rl.py`
- `src/abide/forms/catalog.py`
- `src/abide/training/prime_rl_env.py`
- `tests/integration/`

## Acceptance Criteria

- The modern `prime-rl` path must expose explicit short-form and long-form
  curated subsets instead of hiding length policy in shell presets.
- The Gemma 4 E2B runner must provide realistic canary profiles whose token and
  sequence budgets match the selected subset.
- Focused tests must cover the new form-set/profile behavior.
- Both short-form and long-form canaries must be directly validated on the live
  Gemma 4 E2B `prime-rl` path.

## Progress

- Added explicit `well_known_short` and `well_known_long` subsets to the form
  catalog and exposed them through the modern `prime-rl` environment package.
- Reworked `scripts/run_prime_rl_gemma4_e2b.sh` so runner profiles choose
  subset-aware defaults instead of hardcoding the mixed `well_known` set:
  `smoke` now targets the short subset with a realistic short-form budget,
  while `short-canary`, `long-canary`, and `mixed-canary` resolve to explicit
  subset/token/sequence settings.
- Added entrypoint validation in `scripts/train_prime_rl.py` so obviously bad
  runs fail before launch:
  - long-form or mixed runs now reject undersized `max_tokens` / `seq_len`
  - `batch_size` must be divisible by `rollouts_per_example`
- During live validation, fixed two additional modern-path compatibility bugs in
  `src/sitecustomize.py` / `scripts/train_prime_rl.py`:
  - patched `prime_rl.trainer.perf.PerfCounter` for Gemma 4 configs where
    expert-count fields exist but are `None`
  - switched Gemma 4 trainer-side filesystem weight snapshots from
    `safetensors` to `torch` so tied `embed_tokens` / `lm_head` weights no
    longer break the first post-step broadcast
- Added focused regression coverage for the new subsets, runner profiles, entry
  validation, and the perf-counter compatibility shim.

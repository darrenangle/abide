# RF-080: Stabilize Legacy Verifiers Gemma 4 vLLM Weight Sync

- Priority: `P1`
- Status: `done`
- Area: `training reliability`
- Origin: `VERIFIERS-GEMMA4-BARRIER-2026-04-03`

## Problem

The legacy verifiers Gemma 4 path can boot and complete one-step smokes, but
multi-step runs still fail inside live vLLM weight synchronization with
`RuntimeError: Failed to broadcast barrier_id`. The current Abide compatibility
server preserves the upstream fire-and-forget parameter update pattern, which is
likely too optimistic for the current Gemma 4 + vLLM runtime.

## Primary Files

- `scripts/train_grpo.py`
- `src/abide/verifiers_vllm_server.py`
- `tests/integration/test_train_grpo.py`

## Acceptance Criteria

- The legacy Gemma 4 trainer must serialize or otherwise harden parameter-sync
  updates so multi-step runs do not fail on the pynccl barrier path.
- Obvious non-fatal runtime warnings from this path should be reduced where the
  fix is local and low-risk.
- The new compatibility layer should be covered by focused tests.

## Progress

- Added a legacy-client compatibility shim in `scripts/train_grpo.py` that
  serializes live vLLM parameter updates and waits for the server background
  queue to drain after each barrier, instead of letting updates pile up.
- Reduced the compatibility server default in
  `src/abide/verifiers_vllm_server.py` to one concurrent weight update, with an
  opt-out via `ABIDE_VLLM_MAX_CONCURRENT_WEIGHT_UPDATES`.
- Removed the local `torch_dtype` deprecation and tokenizer/config mismatch
  warnings from the legacy load path by switching to `dtype=` and explicitly
  aligning tokenizer IDs onto the model and generation config.
- Added a Gemma 4-specific allocator-warmup compatibility shim plus an early GPU
  memory preflight in `scripts/train_grpo.py`, so constrained machines now fail
  fast with an actionable message instead of dying deep in Transformers model
  load.
- Made `scripts/run_grpo.sh` choose GPUs from current free-memory data instead
  of hardcoding `0/1`, and fail early if the selected training/vLLM GPUs do not
  meet the minimum free-memory floor.
- Added focused regression coverage for the sync shim, allocator-warmup shim,
  runner defaults, and GPU preflight behavior.

## Remaining

- Done: the live multi-step probe no longer dies in the original NCCL barrier
  path. The remaining failure class is that `vf.RLTrainer` hot-syncs packed
  4-bit `bitsandbytes` tensors into vLLM, while vLLM expects dense Gemma 4
  weights. That blocker belongs to the replacement ticket for the modern
  `prime-rl` path rather than continued investment in the legacy trainer.

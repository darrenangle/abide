# RF-071: Add Gemma 4 E4B Training Profile And Dedicated Canary Runner

- Priority: `P2`
- Status: `done`
- Area: `training reliability`
- Origin: `GEMMA4-CANARY-2026-04-02`

## Problem

The repo has older Gemma 2 and Gemma 3/3n run scripts, but no first-class path
for the newly released Gemma 4 edge models. We need a dedicated E4B canary
runner and shared model-profile defaults so Gemma-family stop tokens,
attention/loading settings, and shell-run parameters do not keep drifting in
copy-pasted scripts.

## Primary Files

- `scripts/train_grpo.py`
- `scripts/train_grpo_trl.py`
- `scripts/run_grpo_gemma4_e4b.sh`
- `scripts/model_profiles.py`
- `tests/integration/`

## Acceptance Criteria

- The training code must recognize Gemma 4 models through a shared profile
  helper instead of open-coded string branches.
- The repo must include a dedicated Gemma 4 E4B canary runner using the current
  stable training path.
- The Gemma 4 profile behavior must be covered by tests.

## Resolution

- Added `scripts/model_profiles.py` so the trainers resolve Gemma-family stop
  tokens, model loading kwargs, and Gemma 4 canary defaults from one place.
- Updated `train_grpo.py` and `train_grpo_trl.py` to use the shared profile
  helper instead of open-coded model-name branches.
- Added `scripts/run_grpo_gemma4_e4b.sh` as the dedicated conservative canary
  runner for `google/gemma-4-E4B-it` using the stable TRL path and reward
  telemetry.
- Added integration coverage for Gemma 4 profile defaults and a CLI smoke test
  proving `train_grpo_trl.py --help` no longer requires optional training
  dependencies to be installed first.
